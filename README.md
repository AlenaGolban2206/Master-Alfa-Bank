## Описание задачи

Банк формирует для клиента несколько кредитных предложений (офферов), которые различаются по:
- сроку кредитования
- процентной ставке
- доступному лимиту
- типу кредитного продукта
- условиям обслуживания

**Цель модели** — предсказать вероятность того, что клиент согласится на конкретное предложение.

**Метрика оценки** — ROC-AUC.

---

## Мой подход к решению

Когда я получила данные, я увидела, что это типичная задача бинарной классификации с сильным дисбалансом классов. Мой план был таким:

1. **Понять данные** — посмотреть, что есть, чего нет, где пропуски
2. **Почистить данные** — убрать мусор, заполнить пропуски
3. **Создать новые признаки** — добавить смысла
4. **Выбрать модель** — LightGBM показался лучшим вариантом
5. **Настроить параметры** — чтобы модель не переобучалась
6. **Проверить на валидации** — GroupKFold, потому что данные сгруппированы по заявкам
7. **Отправить результат** — и получить 75.77 на платформе

## 1. Анализ данных

### Что я увидела в данных

Первым делом я посмотрела на данные и увидела несколько важных вещей:

**1. Сильный дисбаланс классов**
- Согласий всего 6.09% (8,846 записей)
- Отказов 93.91% (136,395 записей)
- Соотношение 15.42:1 — это значит, что модель может просто предсказывать "отказ" и получать высокую точность, но это не то, что нам нужно

**2. Много пропусков**
- Некоторые признаки имеют >90% пропусков (например, `overdraft_app_term_max_360` — 96%)
- Это значит, что их нельзя использовать
- Для признаков с 30-50% пропусков я решила создать флаги пропусков

**3. Категориальные признаки**
- `db_group_last` — тип последнего продукта
- `fl_adminarea` — регион
- Их нужно было закодировать в числа

### Что я сделала с этим

1. **Удалила признаки с >90% пропусков** — они не несут информации
2. **Создала флаги пропусков** для важных признаков с 30-50% пропусков
3. **Заполнила пропуски** медианой (для чисел) и модой (для категорий)
4. **Обработала выбросы** в `offered_rate` через winsorization (обрезала крайние 1%)

## 2. Feature Engineering

### Какие признаки я создала и зачем

Я добавила 5 новых признаков, которые отражают бизнес-логику:

| Признак | Формула | Бизнес-смысл |
|---------|---------|--------------|
| `limit_ratio` | `loan_amount_last / overdraft_limit_max` | Показывает, насколько запрос близок к максимуму. Если близко к 1 — клиент просит почти все, что может дать банк |
| `rate_spread` | `offered_rate - cb_rate` | Премия банка относительно ключевой ставки. Чем меньше, тем привлекательнее предложение |
| `bank_willingness` | `overdraft_limit_max - loan_amount_last` | Готовность банка дать больше, чем просит клиент. Положительное значение — банк готов дать больше |
| `solvency` | `balance_rur_amt_30_min / loan_amount_last` | Платежеспособность. Показывает, сколько на счету относительно запрошенной суммы |
| `credit_activity` | `cnt_cred_loan_90 + cnt_deb_loan_90` | Общая кредитная активность. Больше активности — больше опыта |

### Почему именно эти признаки

Я смотрела на корреляцию с целевой переменной и бизнес-смысл:
- `rate_spread` — ставка важна для клиента
- `bank_willingness` — банк может дать больше, это привлекательно
- `solvency` — если у клиента есть деньги на счету, он платежеспособен

Эти признаки помогли модели лучше понять поведение клиента.

## 3. Борьба с дисбалансом классов

### Что я сделала

Дисбаланс 15:1 — это серьезно. Если просто обучить модель, она будет предсказывать "отказ" почти всегда.

Я использовала параметр `scale_pos_weight = 15.42` в LightGBM. Это увеличивает вес редкого класса (согласия) в 15.42 раза.

```python
scale_pos_weight = len(y[y==0]) / len(y[y==1])  # = 15.42
```

**Почему это сработало**

Модель начала "обращать внимание" на случаи согласия, а не игнорировать их. Это дало прирост ROC-AUC.

## 4. Валидация

### Почему GroupKFold, а не обычный train_test_split

В данных есть заявки (`front_id`), у которых может быть несколько предложений. Обычное случайное разделение может разорвать заявку между train и val — тогда модель увидит в валидации то, что уже видела в обучении, и мы переоценим качество.

**GroupKFold** гарантирует, что все строки одной заявки попадают только в один фолд.

```python
gkf = GroupKFold(n_splits=5)
for train_idx, val_idx in gkf.split(X, y, groups=groups):
```

### Результат валидации

| Фолд | AUC |
|------|-----|
| 1 | 0.8290 |
| 2 | 0.8293 |
| 3 | 0.8227 |
| 4 | 0.8250 |
| 5 | 0.8323 |
| **Средний** | **0.8277** |

Средний AUC 0.8277 — хороший результат.

## 5. Выбор модели и настройка параметров

### Почему LightGBM

Я пробовала несколько моделей (XGBoost, CatBoost, Random Forest), но LightGBM показал лучший результат по скорости и качеству.

**Почему LightGBM:**
- Быстро обучается
- Есть встроенная поддержка дисбаланса (`scale_pos_weight`)
- Хорошо работает с табличными данными
- Устойчив к выбросам

### Как я подбирала параметры

Сначала я использовала Optuna для автоматического подбора. Это дало хороший локальный AUC (0.8218), но на платформе результат был хуже (75.37) — модель переобучилась под валидацию.

Тогда я вернулась к ручному подбору на основе:
- Корреляционного анализа
- Grid Search
- Экспериментов с параметрами

### Финальные параметры

| Параметр | Значение | Почему так |
|----------|----------|------------|
| `n_estimators` | 800 | Хороший баланс качества и времени |
| `learning_rate` | 0.02 | Достаточно низкий, чтобы не переобучаться |
| `num_leaves` | 31 | Оптимальное количество листьев |
| `max_depth` | 7 | Не дает дереву слишком глубоко расти |
| `subsample` | 0.8 | Используем 80% данных для каждого дерева — снижает переобучение |
| `colsample_bytree` | 0.8 | Используем 80% признаков — снижает переобучение |
| `reg_alpha` | 0.05 | L1-регуляризация |
| `reg_lambda` | 0.05 | L2-регуляризация |

### Почему разрыв между CV и платформой

Разрыв есть, и я понимаю почему:

| Причина | Объяснение |
|---------|------------|
| 100% новых заявок в тесте | Test полностью состоит из заявок, которых не было в train |
| Разное распределение target | Доля согласий в тесте может отличаться |
| Размер выборки | Тест гораздо меньше (~36k против 145k в train) |

Лучшее решение — **LightGBM с ручным подбором параметров и GroupKFold валидацией**.

## 10. Что я вынесла из этого опыта

1. Автоматический подбор (Optuna) не всегда лучше ручного — он может переобучиться под валидацию

2. GroupKFold критически важен — без него оценка была бы слишком оптимистичной

3. Признаки активности клиента важнее, чем параметры предложения

4. Дисбаланс классов нужно учитывать обязательно — scale_pos_weight работает

5. Удаление шумовых признаков (>90% пропусков) улучшает качество

6. Иногда простота побеждает — меньше признаков, проще параметры, но стабильнее результат
